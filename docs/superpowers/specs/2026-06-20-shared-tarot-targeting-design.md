# 共享塔罗目标选择设计

## 目标

修复“所有需要目标的塔罗牌均按最高点数选择目标”的策略错误。开包阶段
(`BoosterAgent`) 与背包消耗品使用阶段 (`ConsumableAgent`) 必须使用同一套选择逻辑。

本次只处理目标选择，不改变塔罗牌本身的购买、开包、使用时机或全局动作排序。

## 非目标

- 不引入 checkpoint 搜索、MCTS 或通用决策树。
- 不改动商店购买评分或 genome。
- 不新增单元测试（按用户要求）。
- 不修改无目标塔罗牌的使用逻辑。

## 设计

新增一个纯策略模块，提供按塔罗牌 key 分派的目标选择入口：

```python
choose_tarot_targets(tarot_key, state) -> TarotTargetChoice | None
```

`TarotTargetChoice` 包含提交给 BalatroBot 的目标索引，以及用于日志的选择理由和评分分解。
如果当前局面不存在有正向价值且合法的目标，返回 `None`；调用方不为该塔罗牌生成
候选动作。不会再以“最高点数”作为跨效果的默认兜底。

`BoosterAgent` 和 `ConsumableAgent` 都调用这个入口。两者保留各自的卡牌价值评分与
动作格式：前者生成 `pack`，后者生成 `use`；目标牌索引和理由完全由共享模块产生。

## 目标效果分组

### 单牌/多牌增强

`Magician`、`Lovers`、`Empress`、`Chariot`、`Justice`、`Devil`、`Hierophant`：
优先选择预计会高频出牌且与当前 Joker、主力牌型、已有增强不冲突的牌，而非单纯高点数。
若没有已明确的主力构筑，则保守地选择高点数未增强牌。

### Strength

选择能提升主力牌型形成概率或关键重复价值的牌。其评分优先考虑与已有对子、顺子、
目标点数 Joker 的关系；不只选择最高点数。

### Death

选择一个低价值、与构筑不协同的被覆盖目标，以及一个高价值的复制模板；目标顺序严格
遵循 BalatroBot 的 `pack/use` API 语义。实现前以实际执行日志确认参数顺序，禁止凭
假设固定方向。

### 花色转换

`Sun`、`Moon`、`World`、`Star`：仅当目标花色能支持已确定的 Flush/花色 Joker 构筑时
生成候选。优先转换低价值且不属于目标花色的牌，以增加目标花色密度；没有构筑收益时
不使用。

### Hanged Man

选择最应从牌组移除的牌：低点数、偏离主力牌型、与 Joker 条件冲突或低频出牌的牌。
避免删除高频主力牌、已有高价值增强牌，或会破坏已建立牌型密度的牌。

## 可观测性

对生成的目标塔罗提案，`reasons` 包含塔罗 key、目标索引和核心依据。无法生成候选时，
两个调用点都不提交该动作；本次不扩展 JSONL schema 记录所有未生成原因。

## 验收标准

- 两个调用入口不再各自维护 `_best_tarot_targets`。
- 每个已支持的目标塔罗均按效果类别走专用逻辑。
- `Hanged Man` 不会因通用高点数排序而优先删除最高点数牌。
- `Death` 的目标方向由实际 API 行为确认。
- 全量既有测试继续通过；不新增单测。
