01 / 已验证

胜局路线：一条可复现基线，而不是“已稳定通关”

AGENT1 · RED · WHITE · fixed seed

把经济带过 Boss，再重建 Campfire

这条研究路线把 Delayed Gratification、Joker Stencil 的空槽、Campfire 和跨 Boss 保留现金串成一个前后衔接的经济—得分链。它不是默认策略的统计结论，而是用于后续对照的固定 seed 实验基线。

  1. 1
    保留 Delayed早期不为临时购买过早卖出经济 Joker，让资金增长到足以支持后续路径。
  2. 2
    保留 Stencil 空槽空 Joker 槽维持可跨 Boss 的倍率价值，而不是把槽位机械填满。
  3. 3
    用 Campfire 分段投入出售提供局内 X 倍率，但 Boss 后会重置，不能把 ante 7 的出售数当作 ante 8 的战力。
  4. 4
    现金跨 Boss 留存在 ante 5/6、ante 7 留下足够现金，使 ante 8 商店能恢复出售/商店操作空间。

证据边界

两次胜局，不等于稳定胜率

264从全新 MENU 独立复现的完整步数

两次运行都以 GAME_OVER / won: true 结束。独立复现的 ante 8 前两盲为 55264/50000（Small Blind）与 83556/75000（Big Blind）。

还没有通过 dev / regression / heldout 的完整验证；固定的现金阈值可能已对 AGENT1 的商店序列过拟合。默认 agent 也不能因此被表述为已经稳定通关。

证据与来源（高置信度）

胜局与独立复现：research/findings.md（2026-06-11 条目）和 research/runs/2026-06-11.md。路线的实验基线与“先动态预算、再 regression/heldout”的限制：research/decisions.md（2026-06-11）。其中记录一次续跑胜局与一次从全新状态出发的 264 步复现；两者均返回 won: true。

02 / 运行时

决策链：从游戏快照到可审计动作

每一步都由当前 gamestate 驱动

运行时不直接“读经验文本再自由发挥”。它先把 BalatroBot 的原始快照包装成 GameState，由各阶段 agent 提议动作，校验后选出一个动作；Runner 执行并追加 JSONL 记录。

代码锚点与行为说明

balatro_agent/model.py 定义 GameState、ActionProposal 和 Decision；balatro_agent/agents/__init__.py 组装阶段 agent 并生成提案；balatro_agent/orchestrator.py 调用 validate_action 并按 (score, confidence) 选择；balatro_agent/actions.py 给出合法性规则；balatro_agent/runner.py 执行、在传输异常时确认状态变化并写 JSONL。

03 / 可选分支

checkpoint beam：用真实分支复核候选，但成本仍高

可选 planner

不是每一步都搜索

未挂载 planner 时，Runner 直接执行基线启发式选择。搜索被禁用、只有一个候选，或根 checkpoint 保存失败时，也保留编排器的基线动作。

根状态恢复失败或指纹不一致会抛出 SearchStateMismatch：只有 Runner.run 捕获 SearchStateMismatch 并以 infra_error 返回；Runner.step 会将该异常传播给调用方。

一候选一分支，最后回到根状态

保存 root checkpoint
load 候选分支 A
执行候选 → bounded rollout（按 phase horizon）→ 记录此分支的 StateValue
load 候选分支 B…
比较最佳动作并 restore root

StateValue 以不可协商的字典序比较：outcome → ante → round → blind completion → resources。因此胜负和推进优先于资源；资源再综合剩余手牌、弃牌、金钱、Joker 与消耗品。

outcomeanteroundblind completionresources

当前吞吐边界：完整配置下，6 分支手牌决策约需 35–85 秒，3 候选商店决策约 50 秒；所以它已能验证分支，却还不适合廉价地覆盖完整 cohort。

证据与来源（代码 + 真实运行）

balatro_agent/search.py 实现 root 保存、逐候选 load/execute/rollout、StateValue、根恢复和失败计数禁用。真实成本与动作已生效但传输响应关闭的观察，记录于 research/findings.md 的 2026-06-14 条目；该条目以 balatro_agent/runner.py 的状态变化确认逻辑交叉核对。

05 / 能力边界

记忆如何发挥作用，以及当前没有什么能力

直接影响

已写入 Python 或 genome 的规则

规则只需明确实现并被运行时加载，就会直接影响本局的 ActionProposal 与评分。例如阶段动作、出牌/弃牌评分、商店购买与经济预算的代码规则，或 genome 的评分权重。单元测试与 cohort 评估用于验证、回归门禁和晋升，不是本局动作的前置条件。

直接作用者是运行时 Python / genome；不是 Markdown 文件本身。

间接影响

研究记忆指导下一次假设与实现

研究者或研究 agent 从日志、replay 和 Markdown 中提取可验证问题，选择下一轮实验，并决定哪些证据足以转化为代码或 genome 改动。它帮助避免忘记已知反例，也把固定 seed 发现送往 cohort 验证。

这是一个有人参与、可审计的研究循环，不是隐式“自我学习”。

当前不存在

没有运行时 Markdown 检索、RAG 或在线自我修改

当前 Python 不读取 Markdown 来即时改变本局动作；页面也不应暗示自动 RAG、自动在线学习，或 agent 在游戏进行中自行改写策略代码。Markdown 的作用是保存证据和指导下一次人工/研究 agent 决策。

“记录了经验”与“运行中自动使用经验”是两件不同的事。

为什么要这样划线

README.md 说明策略与研究知识沉淀在 strategy/、research/，Python 负责适配、校验、执行、日志和评估；strategy/README.md 规定稳定策略才考虑转为 Python 规则或 genome 权重；strategy/runs/README.md 定义从 JSONL/replay 到 findings/strategy、再到 dev/regression/heldout 的流程；research/README.md 定义持久研究层。运行时模块的代码路径没有读取这些 Markdown 文件的接口。

01 · 复跑 AGENT1

多次从新局复跑已知路线，分开记录胜负、步数、现金与终局字段，避免只用两局推断概率。

02 · 完成门禁

在 dev 观察后，完成 regression 与 heldout；检查 lost win、分数退化与错误数。

03 · 量化 Lucky Card

逐手对账现金变化，验证或否定已观察到的 Lucky Card 波动，不能将单局差异直接归因于策略。

04 · 降低搜索成本

缩短 rollout、减少高成本候选或复用场景，使 checkpoint 搜索可以承受更多真实评估。
